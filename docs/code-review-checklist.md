# 코드 리뷰 체크리스트

## 기능 요구사항

- [ ] 요구사항을 올바르게 구현했는가?
- [ ] 엣지 케이스를 처리했는가?
- [ ] 기존 기능에 영향을 주지 않는가? (사이드 이펙트 확인)

## 코드 품질

### 가독성

- [ ] 변수/함수 이름이 의도를 명확히 표현하는가?
- [ ] 함수가 하나의 책임만 담당하는가?
- [ ] 복잡한 로직에 적절한 주석이 있는가?
- [ ] 매직 넘버 대신 상수를 사용했는가?

### 중복

- [ ] 중복 코드가 없는가?
- [ ] 기존 유틸리티/헬퍼를 활용했는가?
- [ ] 불필요한 추상화를 만들지 않았는가?

### 복잡도

- [ ] 함수의 길이가 적절한가? (50줄 이내 권장)
- [ ] 중첩 깊이가 3단계 이하인가?
- [ ] 조건 분기가 과도하지 않은가?

## 타입 안전성

- [ ] `any` 타입을 사용하지 않았는가?
- [ ] 타입 단언(`as`)을 최소화했는가?
- [ ] null/undefined 처리가 적절한가?
- [ ] 제네릭을 적절히 활용했는가?

## 테스트 (TDD)

- [ ] 구현 코드보다 테스트 코드가 먼저 작성되었는가? (TDD 사이클 준수)
- [ ] 새 기능에 대한 테스트가 포함되었는가?
- [ ] 기존 테스트가 깨지지 않았는가?
- [ ] 엣지 케이스에 대한 테스트가 있는가?
- [ ] 테스트가 독립적으로 실행 가능한가?
- [ ] 보안 관련 테스트가 포함되었는가? (권한 검증, 입력 검증 등)

## 보안

- [ ] 사용자 입력을 적절히 검증했는가?
- [ ] SQL 인젝션, XSS 등의 취약점이 없는가?
- [ ] 인증/인가가 올바르게 적용되었는가?
- [ ] 시크릿/민감 정보가 코드에 포함되지 않았는가?
- [ ] 에러 메시지에 내부 구현 세부사항이 노출되지 않는가?

## Supabase

- [ ] 새 테이블에 RLS가 활성화되어 있는가?
- [ ] RLS 정책이 요구사항에 맞게 설정되어 있는가?
- [ ] `{ data, error }` 패턴에서 `error`를 확인하는가?
- [ ] `select("*")` 대신 필요한 컬럼만 선택했는가?
- [ ] `SERVICE_ROLE_KEY`가 클라이언트 코드에서 사용되지 않는가?
- [ ] 마이그레이션 파일에 RLS 정책이 포함되어 있는가?
- [ ] `database.ts` 타입 파일이 최신 스키마와 일치하는가?

## 성능

- [ ] 불필요한 리렌더링이 발생하지 않는가?
- [ ] N+1 쿼리 문제가 없는가?
- [ ] 대용량 데이터 처리 시 페이지네이션을 적용했는가?
- [ ] 메모이제이션이 적절히 사용되었는가? (과도한 사용도 확인)
- [ ] 불필요한 API 호출이 없는가?

## 접근성

- [ ] 시맨틱 HTML을 사용했는가?
- [ ] 이미지에 적절한 alt 텍스트가 있는가?
- [ ] 키보드로 모든 인터랙션이 가능한가?
- [ ] 색상 대비가 충분한가?
- [ ] 스크린 리더 지원이 적절한가?

## 스타일 및 컨벤션

- [ ] 프로젝트 네이밍 컨벤션을 따르는가?
- [ ] Biome 린트/포매팅 규칙을 통과하는가?
- [ ] import 순서가 올바른가?
- [ ] 파일 위치가 프로젝트 구조에 맞는가?

## 에러 처리

- [ ] 예외 상황을 적절히 처리했는가?
- [ ] 사용자에게 의미 있는 에러 메시지를 제공하는가?
- [ ] 에러 로깅이 적절한가?
- [ ] API 에러 응답이 일관된 형식인가?

## PR 형식

- [ ] PR 제목이 변경 사항을 간결하게 설명하는가?
- [ ] PR 본문에 변경 사항 요약이 있는가?
- [ ] 관련 이슈가 연결되어 있는가?
- [ ] UI 변경 시 스크린샷이 포함되어 있는가?
- [ ] 불필요한 파일이 포함되지 않았는가? (디버그 코드, console.log 등)
