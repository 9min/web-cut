# 커밋 컨벤션

## 형식

Gitmoji + Conventional Commits 형식을 사용한다.

```
<이모지> <타입>: <제목>

[본문]

[푸터]
```

## 타입별 이모지 매핑

| 이모지 | 타입 | 설명 |
|--------|------|------|
| ✨ | `feat` | 새로운 기능 추가 |
| 🐛 | `fix` | 버그 수정 |
| 📝 | `docs` | 문서 변경 |
| 💄 | `style` | UI/스타일 변경 (코드 로직 변경 없음) |
| ♻️ | `refactor` | 리팩토링 (기능 변경 없음) |
| ✅ | `test` | 테스트 추가 또는 수정 |
| 🔧 | `chore` | 설정 파일, 빌드 스크립트 등 변경 |
| 🚀 | `perf` | 성능 개선 |
| 🔥 | `remove` | 코드/파일 삭제 |
| 🚚 | `rename` | 파일/폴더 이동 또는 이름 변경 |
| 🎨 | `format` | 코드 포맷팅 (기능 변경 없음) |
| 🔒 | `security` | 보안 관련 변경 |
| ⬆️ | `deps` | 의존성 업데이트 |
| 🏗️ | `arch` | 아키텍처 변경 |
| 🗃️ | `db` | 데이터베이스 관련 변경 |
| 🔀 | `merge` | 브랜치 머지 |

## 제목 작성 규칙

- 한국어로 작성한다.
- 50자 이내로 간결하게 작성한다.
- 마침표를 붙이지 않는다.
- 명령형이 아닌 서술형으로 작성한다.

### 좋은 예

```
✨ feat: 사용자 로그인 기능 추가
🐛 fix: 토큰 만료 시 리다이렉트 오류 수정
♻️ refactor: API 클라이언트 구조 개선
✅ test: 로그인 서비스 단위 테스트 추가
```

### 나쁜 예

```
feat: fix bug              # 타입과 설명이 불일치
✨ feat: 수정.              # 마침표 사용, 내용 불명확
✨ feat: 사용자가 로그인할 수 있도록 이메일과 비밀번호 입력 폼을 추가하고 API 연동   # 너무 김
```

## 본문 작성 규칙

- 제목만으로 설명이 부족한 경우에 작성한다.
- **무엇**을 **왜** 변경했는지 설명한다.
- 72자 단위로 줄바꿈한다.

```
✨ feat: 소셜 로그인 기능 추가

Google, GitHub OAuth 로그인을 지원한다.
기존 이메일 로그인과 동일한 사용자 플로우를 유지하되,
신규 사용자의 경우 자동 회원가입 처리한다.
```

## 푸터 작성 규칙

- 관련 이슈 번호를 참조한다.
- Breaking Change가 있는 경우 명시한다.

```
✨ feat: 인증 미들웨어 구조 변경

토큰 검증 로직을 미들웨어로 분리한다.

BREAKING CHANGE: authMiddleware 인터페이스 변경
Closes #42
```
